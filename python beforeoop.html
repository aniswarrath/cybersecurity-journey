Dictionaries — Complete Reference

What a Dictionary Is

A dictionary stores data as key-value pairs. Every value has a unique key you use to access it. Like a real dictionary — word is the key, definition is the value. Unlike lists which use index numbers, dictionaries use meaningful keys.

# Basic dictionary
device = {
    "ip": "192.168.1.1",
    "port": 22,
    "protocol": "TCP",
    "status": "open"
}

# Access by key
print(device["ip"])        # 192.168.1.1
print(device["port"])      # 22

Why .get() Is Better Than Direct Access

# Direct access — crashes if key missing
services[9999]   # KeyError: 9999 — program crashes

# .get() — never crashes
services.get(9999, "Data not available")   # returns "Data not available"

# How .get() works internally — equivalent to:
if 9999 in services:
    return services[9999]
else:
    return "Data not available"
Q: How does .get() know what to do when key is missing?
A: You literally tell it with the second argument. services.get(port, "Data not available") means: look for port as a key — if found return its value — if NOT found return "Data not available". The second argument IS the instruction for the missing case. Python doesn't guess — you define the fallback.

Nested Dictionaries

A dictionary where the values are themselves dictionaries. Two levels deep — outer key gets inner dictionary, inner key gets the actual value.

network_scan = {
    "192.168.1.1": {"port": 80, "status": "open"},
    "192.168.1.2": {"port": 22, "status": "open"},
    "192.168.1.3": {"port": 3306, "status": "closed"}
}

# Access nested value — two levels of indexing
print(network_scan["192.168.1.1"]["port"])     # 80
print(network_scan["192.168.1.1"]["status"])   # open

# Loop through nested dictionary
for ip, value in network_scan.items():
    print(f"{ip} — port {value['port']} — {value['status']}")

Nested Loop + Nested Dictionary — Multiple Ports Per Device

network = {
    "192.168.1.1": {80: "HTTP", 443: "HTTPS", 53: "DNS"},
    "192.168.1.19": {22: "SSH", 8080: "HTTP-ALT"},
    "192.168.1.30": {21: "FTP", 3306: "MySQL"}
}

for ip, ports in network.items():
    print(f"\n{ip}")
    for port, service in ports.items():
        print(f"  port {port} — {service}")
Q: Why don't we use ['service_name'] when looping through the second dict?
A: Because the port number IS the key and the service name IS the value. When you write for port, service in ports.items() — port gets the key (80) and service gets the value ("HTTP") directly. No need to index by name because the data IS the key and value themselves. Different from the nested dict where inner keys had explicit names like "port" and "status".

ports = [80, 443, 22, 21, 3306, 8080, 9999, 135, 25, 53]

services = {

Error Handling — try · except · finally · raise

Why Error Handling Exists

Without error handling — any unexpected input crashes your program with an ugly traceback. With error handling — you catch problems gracefully and keep the program running. The difference between if/else and try/except: if/else checks conditions BEFORE code runs. try/except catches crashes DURING code running — things you cannot predict in advance.

Complete Structure

try:
    # risky code here
except ValueError:
    # runs if ValueError occurs
except ZeroDivisionError:
    # runs if ZeroDivisionError occurs
except Exception as e:
    # catches ANY other error — e tells you what went wrong
    print(f"Unexpected error: {e}")
else:
    # runs ONLY if NO error occurred
finally:
    # runs ALWAYS — error or not — use for cleanup

Built Exercise — Division with Error Handling

try:
    number = int(input("Enter a number: "))   # ValueError if text entered
    result = 100 / number                      # ZeroDivisionError if zero
    print(f"100 / {number} = {result}")
except ValueError:
    print("Error — please enter a number not text")
except ZeroDivisionError:
    print("Error — cannot divide by zero")
finally:
    print("Program finished — this always runs")

# Test inputs:
# 5     → 100 / 5 = 20.0 → Program finished
# hello → Error — please enter a number → Program finished
# 0     → Error — cannot divide by zero → Program finished
# #     → Error — please enter a number → Program finished (# is a string)
Q: If if/else is sufficient — why does try/except exist?
A: if/else checks conditions BEFORE code runs — things you anticipate. try/except catches crashes DURING code running — things you cannot predict. You cannot write an if statement to handle every possible wrong input. try/except handles anything that crashes at runtime — no matter what it is.
Q: Does # count as text for ValueError?
A: Yes. Everything from input() comes in as a string — letters, numbers, symbols, spaces — all strings. int("#") tries to convert # to integer — fails — ValueError. Python doesn't distinguish between # and "hello" — both are just strings that cannot become integers.

finally — Always Runs — Cleanup

# Without finally — socket might stay open if error occurs
sock = socket.socket()
try:
    result = sock.connect_ex((ip, port))
except Exception as e:
    print(f"Error: {e}")
sock.close()   # might never run if error above

# With finally — socket ALWAYS closes no matter what
sock = socket.socket()
try:
    result = sock.connect_ex((ip, port))
except Exception as e:
    print(f"Error: {e}")
finally:
    sock.close()   # guaranteed to run always

raise — Manually Trigger an Error

def check_username(username):
    if username[0].isdigit():
        raise ValueError("Username cannot start with a number")
    if len(username) < 3:
        raise ValueError("Username too short — minimum 3 characters")
    return True

try:
    check_username("123anish")
except ValueError as e:
    print(f"Invalid username: {e}")

Sockets — Python Network Programming

What a Socket Is

A socket is an endpoint for sending and receiving data over a network. Every network connection ever made — HTTP, SSH, FTP, SMTP — uses sockets underneath. When your browser visits google.com — it creates a socket, connects to Google's IP on port 443, sends an HTTP request, receives the response, closes the socket.

check_port Function — Line by Line

import socket

def check_port(ip, port):
    try:
        # Create TCP/IPv4 socket — pick up the phone
        sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)

        # Set timeout — don't wait more than 1 second
        sock.settimeout(1)

        # Try to connect — dial the number
        result = sock.connect_ex((ip, port))

        # Hang up — always close
        sock.close()

        # result == 0 means connection succeeded — port OPEN
        # result != 0 means connection failed — port CLOSED or FILTERED
        return result == 0

    except Exception as e:
        return False   # any unexpected error — treat as closed

connect vs connect_ex

MethodOn failureUse in port scanner
sock.connect()Raises exception — crashesNo — crashes on first closed port
sock.connect_ex()Returns error code — continuesYes — handles closed ports gracefully
Q: Why does port scanner need settimeout(1)?
A: Without timeout — scanner hangs forever on filtered ports where firewall silently drops packets and never responds. One filtered port = scanner freezes completely. With settimeout(1) — waits maximum 1 second — moves on — scan completes in reasonable time.

DNS Lookup with Socket

import socket

ip = socket.gethostbyname("google.com")
print(f"google.com → {ip}")    # google.com → 142.251.222.142
Library network blocking: When testing at the library — all ports returned False including Google's port 80 and 443. This is egress filtering — the library firewall blocks outgoing connections. Your code was correct — the network was the variable. At home — ports 80 and 443 on Google return True.
