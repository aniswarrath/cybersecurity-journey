Part 12 — SSH — Secure Shell

What SSH Is

SSH — Secure Shell — is a cryptographic network protocol that provides secure encrypted remote terminal access to servers over an unsecured network. It replaced Telnet — the old unencrypted remote access protocol — where everything including passwords traveled as plain text. SSH encrypts everything end to end.

FeatureTelnet — oldSSH — modern
EncryptionNone — plain textFull encryption end to end
Password securityVisible on networkEncrypted — unreadable
Port2322
StatusDead — never useStandard — used everywhere
Q: What is a cryptographic network protocol?
A: Three words — Cryptographic = uses encryption and mathematical algorithms to secure data. Network = works over a computer network. Protocol = a set of rules two computers agree to follow when communicating. Together: a set of rules for two computers to communicate over a network where all communication is mathematically encrypted.
Q: What is an unsecured network?
A: Any network where traffic can be intercepted and read by others. The internet itself is unsecured — your data passes through dozens of routers, ISPs, and networks you don't control. Public WiFi at a cafe — anyone on same network can capture packets. SSH encrypts everything before it leaves your device — even if someone captures your packets — they see only gibberish.

SSH Authentication — Two Methods

Method 1 — Password Authentication

You connect and type a password. Password travels encrypted to server. Server checks if correct. Access granted. Simple but vulnerable to brute force — attacker tries thousands of passwords automatically.

Method 2 — Key-Based Authentication — More Secure

You generate a mathematically linked key pair. Private key stays on your laptop — never sent anywhere — ever. Public key is placed on the server. When you connect — server sends a challenge — your private key solves it mathematically — server verifies with public key — access granted. No password ever sent — nothing to brute force.

FeaturePassword AuthKey-Based Auth
What is sentEncrypted passwordMathematical proof — private key never sent
Brute force riskYes — attacker tries thousands of passwordsNo — 2^256 possibilities — impossible
Phishing riskYes — trick you into typing on fake siteNo — private key never typed or transmitted
Security levelMediumVery high
Q: What is a private key and why does it never need to leave your laptop?
A: A private key is a very large mathematically generated secret number. When you connect to a server — the server sends a challenge — a puzzle only your private key can solve. You solve it and send the solution back. Server verifies using your public key. You proved you have the private key WITHOUT ever sending it. The private key never leaves your device because it never needs to — only the solution to the challenge is sent — and that solution is useless for future connections.

SSH Keys — What Gets Generated

id_ed25519.pub~/.ssh/id_ed25519.pubPublic key — goes on servers you want to accessYes — safe to share
ed25519 is the encryption algorithm — modern elliptic curve cryptography. Faster and more secure than older RSA keys. The -t flag in ssh-keygen specifies the algorithm type.

Q: If attacker gets private key file — do they need your Mac password to use it?
A: No. Attacker copies the key file to their own laptop and uses it from there: chmod 600 stolen_key → ssh -i stolen_key apple@yourserver.com. They don't need your Mac at all. This is why passphrase on the key is critical — even if key file is stolen — useless without passphrase.

Part 13 — FTP — File Transfer Protocol

What FTP Does

FTP — File Transfer Protocol — is a protocol designed specifically for transferring files between computers over a network. That is its only job — sending and receiving files. You connect to an FTP server, log in with username and password, browse files, and upload or download.

FTP Uses Two Ports

PortPurpose
21Control channel — sends commands — login, list files, navigate directories
20Data channel — actual file transfer happens here

The Massive Security Problem with FTP

FTP is completely unencrypted — like HTTP. Everything travels as plain text. Anyone on the same network running Wireshark can capture your FTP credentials and every file you transfer. Username — visible. Password — visible. Every file — visible.

Finding port 21 open during a penetration test = immediate critical finding. Recommendation: disable FTP entirely and use SFTP instead.

FTP Replacements — Always Use These Instead

ProtocolFull nameHow it worksPortUse
SFTPSSH File Transfer ProtocolCompletely different protocol — runs inside SSH tunnel — not FTP at all despite similar name22Always prefer this
FTPSFTP SecureOriginal FTP with TLS encryption added on top — like HTTPS is HTTP with TLS990Acceptable if SFTP not available
SCPSecure Copy ProtocolSimple encrypted file transfer via SSH — good for quick transfers22Quick secure file copies
SFTP is NOT FTP with encryption. SFTP is a completely separate protocol that happens to transfer files. It was designed from scratch as part of SSH. FTPS is actual FTP with TLS added. Different protocols — different design — same result of secure file transfer.

Port Scanner — FTP Findings

Port foundServiceSecurity action

22 openSSH and SFTPCheck auth method — disable password auth

Part 14 — SMTP — Simple Mail Transfer Protocol

What SMTP Does

SMTP — Simple Mail Transfer Protocol — is the protocol responsible for sending email from one server to another. When you hit Send on Gmail — SMTP carries that email to the recipient's mail server. SMTP only sends — it does not receive.

Three Email Protocols — Know the Difference

POP3Post Office ProtocolDownloads email then deletes from serverIncoming

Three Email Security Standards — SPF, DKIM, DMARC

StandardFull nameWhat it doesProtects against
SPFSender Policy FrameworkDNS record listing which servers are allowed to send email for your domainEmail spoofing — attacker pretending to be razorpay.com
DKIMDomainKeys Identified MailDigitally signs every outgoing emailEmail tampering — verifies email not modified in transit — Integrity

You saw this in your Razorpay recon: nslookup -type=MX razorpay.com returned aspmx.l.google.com — Google's SMTP servers. Razorpay uses Google Workspace which includes full SPF, DKIM, DMARC protection. This means phishing attacks against Razorpay employees must bypass Google's spam filters — not just Razorpay's own defenses.

Email Spoofing — Security Attack

Attacker sends email pretending to be from razorpay@razorpay.com. Without SPF, DKIM, DMARC — this is trivially easy. With proper configuration — spoofed emails are rejected before reaching the inbox. Finding misconfigured email security = high value finding in penetration tests.

STARTTLS Downgrade Attack

When using port 587 with STARTTLS — connection starts plain text then upgrades to encrypted. Attacker sitting between client and server can intercept before the upgrade happens and force the connection to stay plain text. Defense — use port 465 which is encrypted from the very first packet with no plain text phase.

Part 15 — SSL vs TLS — The Difference

What SSL Was

SSL — Secure Sockets Layer — was the original encryption protocol invented in 1995 by Netscape. It was designed to secure web communications. Over time serious vulnerabilities were discovered in all versions of SSL — POODLE, BEAST, Heartbleed — making it fundamentally broken and unusable.

What TLS Is

TLS — Transport Layer Security — is the modern replacement for SSL. Redesigned from scratch to fix SSL's vulnerabilities. Actively maintained and updated. TLS 1.3 — released 2018 — is the current standard. Faster, simpler, and more secure than all previous versions.

ProtocolYearStatusShould you use?
SSL 1.0, 2.0, 3.01995–1996Dead — broken — multiple critical vulnerabilitiesNever
TLS 1.01999Deprecated — weakNo
TLS 1.12006Deprecated — weakNo
TLS 1.22008Still acceptable with correct configurationAcceptable
TLS 1.32018Current standard — use thisAlways prefer
Why people still say SSL: Habit. The name stuck even though SSL is dead. When someone says "SSL certificate" they actually mean TLS certificate. When your browser shows a padlock — it's using TLS 1.3 — not SSL. The industry uses the terms interchangeably even though they're different things. Always assume TLS when someone says SSL in modern context.

Security Relevance — Checking TLS Version

During a penetration test — one of the first checks is which TLS version a server supports. Server still supporting TLS 1.0 or 1.1 = finding. Server supporting SSL 3.0 = critical finding. Modern servers should only accept TLS 1.2 and 1.3.

Part 16 — Packet Sniffing and Wireshark

What Packet Sniffing Is

Packet sniffing is capturing packets traveling on a network and reading their contents. Like placing a microphone in a room — it records everything passing through. Wireshark is the most widely used free packet capture and analysis tool. Used by security professionals for network analysis and by attackers for credential theft.

How Attackers Use Wireshark

StepWhat happens
1Attacker connects to same WiFi network as victim
2Opens Wireshark — starts capturing all network traffic
3Victim opens http://192.168.1.1 and types admin password
4Wireshark displays: POST /login username=admin&password=mypassword123
5Attacker reads credentials directly — no hacking required
This is a passive attack — completely silent and undetectable. Attacker sends nothing — only listens. No alerts triggered. No logs created on victim's system. This is why HTTP is so dangerous and why HTTPS is mandatory for any login page.

HTTP vs HTTPS — What Wireshark Sees

ProtocolWhat Wireshark capturesAttacker gets
HTTP — port 80Complete plain text — every byte readableUsername, password, session cookies, all content
HTTPS — port 443Encrypted gibberish — x7k#9@mN2$pQ8...Nothing useful — cannot decrypt without key
Q: Can attacker capture data from home after leaving the cafe WiFi?
A: No. Packet sniffing only works on the SAME local network you are currently connected to. Once attacker leaves the cafe — different network — cannot see cafe traffic anymore. The attack must happen while physically on the same network. Attacker uses VPN not to steal data remotely — but to hide their identity while performing the attack at the cafe.

What Router and ISP Can See

What they CAN seeWhat they CANNOT see
Which websites visited — domain namesWhat you did inside HTTPS websites
Time of each visitYour messages, searches, passwords on HTTPS sites
How much data transferredContent of any encrypted page
Which app connected to which serverContent of end-to-end encrypted apps like WhatsApp
VPN changes everything: With VPN active — router and ISP see only encrypted traffic going to VPN server. They cannot see which websites you visit or which apps you use. This is why VPN is essential on public WiFi.

Part 17 — Sockets — Python Network Programming

What a Socket Is

A socket is an endpoint for sending and receiving data over a network. Think of it like a telephone — you need one on each end to have a conversation. Every network connection you've ever made — HTTP, SSH, FTP, SMTP — uses sockets underneath. When your browser visits google.com — it creates a socket, connects to Google's IP on port 80 or 443, sends an HTTP request, receives the response, closes the socket.

Socket Analogy

Phone callSocket equivalent
Pick up handsetsocket.socket() — create socket
Dial phone numberIP address
Dial extensionPort number
Call connectsconnect_ex() — connection established
TalkData being transferred
Hang upsock.close()

Creating a Socket in Python

CodeWhat it means
import socketImport Python's networking module — brings in all socket tools
socket.socket(socket.AF_INET, socket.SOCK_STREAM)Create a TCP socket — AF_INET means use IPv4 — SOCK_STREAM means use TCP
sock.settimeout(1)Wait maximum 1 second — if no response in 1 second — give up and move on
sock.connect_ex((ip, port))Try to connect — returns 0 if success — returns non-zero if failed
sock.close()Close the connection — always close after use — releases resources

connect vs connect_ex — Critical Difference

MethodWhat happens on failureUse in port scanner
sock.connect()Raises exception — program crashes if port closedNo — crashes on first closed port
sock.connect_ex()Returns error code — program continues — 0=open, non-zero=closedYes — handles closed ports gracefully
Why settimeout(1) is critical: Without timeout — scanner waits forever on filtered ports — ports where firewall silently drops packets. A scan of 1000 ports could take hours. With 1 second timeout — maximum 1000 seconds for 1000 ports. In practice much faster since most ports respond immediately.

Complete check_port Function — Line by Line

LinePlain English explanation
sock = socket.socket(AF_INET, SOCK_STREAM)Pick up the telephone handset — get ready to make a TCP call using IPv4
sock.settimeout(1)If nobody answers within 1 second — hang up and move on
result = sock.connect_ex((ip, port))Dial the number — try to call that IP on that port — store what happened
sock.close()Hang up the phone — release connection
return result == 0If result is 0 — someone answered — port OPEN — return True. Otherwise False.

What the Socket Does Under the Hood

When you call connect_ex() — your Python code automatically performs the full TCP three-way handshake — SYN, SYN-ACK, ACK — that you already understand from networking theory. If the handshake completes — result = 0 — port is open. If server sends RST — port closed — non-zero result. If no response — timeout — filtered port.

This is where theory becomes code. Everything you learned about TCP handshake, ports, and networking — your port scanner code does automatically every time you call connect_ex(). You're not doing anything new — you're writing the code that implements what you already understand.

Why Library Network Blocked Socket Connections

When testing at the library — all ports returned False including Google's port 80 and 443. This is called egress filtering — the library's network blocks outgoing connections to certain ports to control internet access and prevent data exfiltration. This is a legitimate security control used in organizations. At home — ports 80 and 443 on Google will return True — your code is correct.

Part 18 — WiFi Security, Defence in Depth, Zero Trust

How WiFi Encryption Works — WPA2 and WPA3

WiFi encrypts the wireless radio signal between your device and router using WPA2 — WiFi Protected Access 2 — or WPA3. This protects the signal from outsiders who are not connected to the network. Someone outside the cafe cannot capture and read your WiFi traffic. However WiFi encryption does NOT protect device-to-device communication inside the network.

Why DHCP Spoofing Works Inside Encrypted WiFi

WiFi password = building entrance security. Once inside the building — everyone can talk to everyone in the corridors. WiFi encryption protects the entrance — not what happens inside. Attacker who knows the WiFi password is already inside — WiFi encryption does nothing to stop them from performing DHCP spoofing, ARP poisoning, or packet sniffing inside the network.

Layers of Protection — Defence in Depth

ThreatProtection layer needed
Outsider intercepting WiFi signalWPA2/WPA3 — WiFi encryption
Insider DHCP spoofingDHCP snooping on managed enterprise switches
Insider ARP poisoningDAI — Dynamic ARP Inspection — on managed switches
Insider packet sniffing HTTPHTTPS — encrypts content end to end
Device-to-device attacksClient isolation — 802.1X — network segmentation
Any network attackVPN — Virtual Private Network — encrypts everything

Zero Trust — The Modern Security Architecture

Zero Trust emerged from exactly the gap identified above — devices inside a network should not automatically trust each other just because they're inside. Traditional security assumed inside = trusted. One breach = attacker has access to everything inside.

Zero Trust core principle: "Never trust, always verify — even devices already inside the network."

Traditional securityZero Trust
Inside network = trusted — outside = not trustedNothing trusted by default — inside or outside
Get past the firewall = access everythingEvery device, every user, every connection verified every time
One breach = entire network compromisedOne breach = limited blast radius — attacker contained
Username + password = full accessContinuous verification — device health, location, behavior
This is in your Phase 4 roadmap. Zero Trust architecture is one of the most in-demand skills in enterprise security. Companies like Google, Microsoft, and Cloudflare have fully adopted Zero Trust. Understanding why it exists — from the ground up as you did — makes you more valuable than someone who just knows the buzzword.

Supply Chain Attacks — Currently Going Viral

Supply chain attacks compromise software or tools before they reach you. Instead of breaking in through the front door — attacker hides malware inside software you choose to install and trust. Your defenses see legitimate software — no alerts triggered.

Real attackWhat happenedImpact

SSH key theft via supply chain: Malicious tool reads your ~/.ssh/id_ed25519 and silently sends it to attacker's server. Your defenses see legitimate software running. No alerts. This is exactly why passphrase on SSH keys matters — even if key is stolen this way — attacker cannot use it without the passphrase.

Q: Is cryptocurrency anonymous?
A: Bitcoin is pseudonymous — not anonymous. Every transaction is permanently recorded on the public blockchain — visible to anyone. Investigators follow the money trail: theft → wallet → exchange → cash out. Cash out points — cryptocurrency exchanges — must comply with KYC — Know Your Customer — regulations. They require identity verification. Privacy coins like Monero are harder to trace but not impossible. The moment money is converted to fiat currency — trail becomes much clearer.
